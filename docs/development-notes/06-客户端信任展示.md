## 修改 `_calc_objective_score()`

找到这里：

```python
        success_rate = (success_count / usage_count) * 100.0
        avg_response_time_ms = sum(response_times) / len(response_times) if response_times else 0.0

        latency_score = self._calc_latency_score(avg_response_time_ms)
        volume_confidence = min((usage_count / 20.0) * 100.0, 100.0)

        objective_score = (
            0.50 * success_rate +
            0.30 * latency_score +
            0.20 * volume_confidence
        )
```

改成下面这版：

```python
        success_rate = (success_count / usage_count) * 100.0
        avg_response_time_ms = sum(response_times) / len(response_times) if response_times else 0.0

        latency_score = self._calc_latency_score(avg_response_time_ms)
        volume_confidence = min((usage_count / 20.0) * 100.0, 100.0)

        # 第一阶段加入轻量样本保护：
        # 样本越少，客观分越向中性区间收缩，避免极少量调用把分数拉得过高或过低
        raw_objective_score = (
            0.50 * success_rate +
            0.30 * latency_score +
            0.20 * volume_confidence
        )

        if usage_count < 5:
            shrink_ratio = usage_count / 5.0
            objective_score = 50.0 * (1 - shrink_ratio) + raw_objective_score * shrink_ratio
        else:
            objective_score = raw_objective_score
```

这个修正的意义是：

- 1 条调用成功，不至于看起来像“超级可靠”
- 1 条调用失败，也不至于直接把服务打死
- 更适合你当前项目体量

# 你很可能会碰到的两个常见问题

我先提前告诉你，方便你定位。

### 问题 1：测试服务 `endpoint_url` 不可用

如果 `test-service` 指向的是假的 `http://example.com/api`，那调用阶段会失败。
这不是评价体系的 bug，而是测试服务本身不可调用。

这种情况下有两种办法：

- 临时把测试服务改成你本地能返回 JSON 的 mock 接口
- 或者先接受“失败调用也会产生 usage”，然后拿这条 `usage_id` 提交 `outcome=fail` 的评价

这其实也能验证失败路径。

### 问题 2：旧测试数据里 `Usage` 的新字段为空

如果历史数据是在你加 `http_mode / final_state / is_meaningful` 之前生成的，旧记录可能没有这些值。

这种情况下，`backfill_trust_stats.py` 可能会让一些服务的历史 usage 统计偏少。
第一阶段这是可以接受的。后面你可以根据需要补一个一次性的 SQL，把历史 usage 的默认值填上。
