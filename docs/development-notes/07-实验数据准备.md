# 第七步：最终收尾

## 这一步只修 3 个问题

### 问题 1：失败调用竟然也能被评价成 `success`

你这次联调里最明显的问题就是：

- 实际调用因为上游 SSL 失败
- `usage.final_state` 客观上是 `fail`
- 但测试脚本仍然提交了 `outcome = "success"`

这在系统语义上是不合理的。
Agent 可以对“上游成功但对任务失败”做降级评价，但**不能把客观失败评价成成功**。

### 问题 2：主观分没有把 `outcome` 纳入计算

现在 `subjective_score` 只看：

- `task_fit`
- `output_quality`
- `protocol_adherence`
- `cost_satisfaction`
- `would_reuse`

但不看 `outcome`。
这会导致“明明失败了，只要其他分打高，主观分还是很高”。

### 问题 3：测试脚本把 `outcome` 写死成 `success`

这会让闭环测试虽然“能跑通”，但语义不严谨。
测试脚本应该根据真实 `usage.final_state` 自动决定本次评价的 `outcome`。

# 修改 1：`app/api/reviews.py`

## 目标

让评价结果必须和真实调用状态一致，或者**比真实状态更保守**。

### 规则建议

- `usage.final_state == success`
  允许评价为：`success / partial / fail`
- `usage.final_state == partial`
  允许评价为：`partial / fail`
- `usage.final_state == fail`
  只允许评价为：`fail`

这很合理，因为：

- 调用成功，Agent 下游仍可能失败，所以允许降级
- 调用失败，不允许硬说成功

------

## 你要改的代码

在 `app/api/reviews.py` 里，找到这段：

```python
    # 7) outcome 合法性校验
    if review_data.outcome not in ["success", "partial", "fail"]:
        raise HTTPException(
            status_code=status.HTTP_400_BAD_REQUEST,
            detail="outcome 仅支持 success / partial / fail"
        )
```

把它替换成下面这段：

```python
    # 7) outcome 合法性与真实调用状态一致性校验
    if review_data.outcome not in ["success", "partial", "fail"]:
        raise HTTPException(
            status_code=status.HTTP_400_BAD_REQUEST,
            detail="outcome 仅支持 success / partial / fail"
        )

    # 第一阶段规则：
    # - success 调用允许评价为 success / partial / fail（允许Agent按下游效果降级）
    # - partial 调用允许评价为 partial / fail
    # - fail 调用仅允许评价为 fail
    allowed_outcomes_map = {
        "success": ["success", "partial", "fail"],
        "partial": ["partial", "fail"],
        "fail": ["fail"]
    }

    allowed_outcomes = allowed_outcomes_map.get(usage.final_state, ["fail"])
    if review_data.outcome not in allowed_outcomes:
        raise HTTPException(
            status_code=status.HTTP_400_BAD_REQUEST,
            detail=f"该调用真实状态为 {usage.final_state}，不允许提交 outcome={review_data.outcome}"
        )
```

替换成下面这版：

```python
            # 先按1~5评分计算组合分，再映射到0~100
            base_score = (
                0.35 * review.task_fit +
                0.30 * review.output_quality +
                0.20 * review.protocol_adherence +
                0.15 * cost_score
            ) * 20.0

            # 是否愿意再次使用作为轻量修正项
            if review.would_reuse:
                base_score += 5.0
            else:
                base_score -= 5.0

            # 将最终结果 outcome 纳入主观分修正
            # 第一阶段采用轻量规则：
            # - success：小幅加分
            # - partial：不加不减
            # - fail：明显扣分
            outcome_adjust_map = {
                "success": 5.0,
                "partial": 0.0,
                "fail": -15.0
            }
            base_score += outcome_adjust_map.get(review.outcome, 0.0)
```

这样做完后：

- `fail` 评价不可能再轻易算出特别高的主观分
- `success` 才会拿到完整正向加成
