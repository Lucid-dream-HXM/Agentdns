**实验基座目录结构与落地实施顺序 v1**。
这一版的目标只有一个：

**把前面已经定好的整套实验设计，变成你现在可以开始动手落地的工程结构和执行顺序。**

这一步之后，你就不再是“有实验想法”，而是有一套明确的实施蓝图。

------

# 一、总体原则

实验基座必须满足四个要求：

1. **不污染主后端业务代码**
2. **和当前 AgentDNS 后端真实接口联动**
3. **可以分阶段逐步跑通**
4. **先做最小闭环，再扩成完整主实验**

所以我不建议把实验代码散落在根目录，也不建议一上来就写大而全脚本。
最合理的是把实验工程单独收进一个统一目录。

# 三、每一层目录到底负责什么

## 1. `configs/`

这是**实验定义层**。

所有你后面会反复改的东西，尽量都放这里，不写死在 Python 代码里。

它负责：

- 平台里有哪些用户、组织、Agent
- 192 个逻辑服务的静态目录
- 服务动态行为参数
- 三阶段漂移覆盖规则
- 种子任务模板
- 6 类 Agent 画像
- 四个消融版本的运行配置

这样后面改实验，不用改业务代码。

------

## 2. `mock_services/`

这是**模拟服务运行层**。

它本质上是一个单独的 FastAPI 小服务，负责对外暴露多类模拟服务 endpoint。
它不是 AgentDNS 本体，而是 AgentDNS 调用的“上游服务世界”。

这一层负责：

- 接收来自 AgentDNS 的 HTTP 调用
- 根据 `service_key` 或 `agentdns_uri` 找到对应行为画像
- 按当前实验阶段生成：
  - success / partial / fail
  - latency
  - schema 是否正确
  - 内容质量隐藏分
- 对外返回看起来像真的业务结果
- 对内写 oracle 日志

这层是整个实验可控性的核心。

------

## 3. `seeds/`

这是**平台初始化层**。

这一层负责：

- 生成用户、组织、Agent、服务目录
- 将它们真正注册进当前 AgentDNS 平台
- 清空并重置实验环境

后面每跑一组正式实验前，都要靠这层重建或重置状态。

------

## 4. `runners/`

这是**实验执行层**。

它直接走当前已经实现好的 HTTP 接口，负责：

- 任务实例化
- discovery 搜索
- Top-K 截断
- Agent 本地选择
- sync/stream/async 调用
- retry / reselection
- review 提交
- 多轮阶段排程
- 四版本配对消融运行

它是“自动化批量实验”的发动机。

------

## 5. `evaluators/`

这是**评估计算层**。

它不负责跑任务，只负责算分。

主要包括：

- 任务是否完成
- oracle utility
- 命中型选择指标
- trust 相关统计指标

这一层必须独立出来，因为后面你很可能要反复调公式和表格。

------

## 6. `analyzers/`

这是**结果汇总与论文出图层**。

它负责：

- 汇总多轮实验结果
- 导出 CSV
- 生成图表
- 抽取典型案例

这一层最后会直接服务于论文写作。

------

## 7. `outputs/`

这是**结果落盘层**。

所有实验产物都要归档在这里，不要散落在根目录。

建议区分：

- 原始运行日志
- oracle 日志
- 聚合结果
- 最终表格
- 最终图

这样后面论文复盘不会乱。

# 四、哪些文件是第一批必须写的

现在最重要的不是把整个目录一次性写完，而是抓出**最小闭环所必需的第一批文件**。

我建议第一批只做下面这些。

## A. 模拟服务最小闭环

必须先有：

```text
experiments/mock_services/app.py
experiments/mock_services/registry.py
experiments/mock_services/oracle_logger.py
experiments/mock_services/handlers/text_generation.py
experiments/mock_services/handlers/structured_extraction.py
experiments/mock_services/utils/random_utils.py
```

为什么先只做两个 handler：

- `text_generation`
- `structured_extraction`

因为这两类最容易先跑通最小闭环：

- 一类开放式文本
- 一类结构化约束任务

它们能覆盖 discovery、call、review、trust 的关键链条。

------

## B. 平台初始化最小闭环

必须先有：

```text
experiments/seeds/generate_platform_entities.py
experiments/seeds/generate_service_catalog.py
experiments/seeds/seed_platform_entities.py
```

这一批先不用上来生成 192 个服务。
第一轮只生成：

- 2 个 provider user
- 2 个 organization
- 6 到 12 个服务
- 2 个 client user
- 4 个 agent

先把最小实验市场做出来。

------

## C. 任务与执行最小闭环

必须先有：

```text
experiments/configs/tasks/task_templates_document.yaml
experiments/configs/agents/agent_profiles.yaml
experiments/runners/task_instantiator.py
experiments/runners/discovery_runner.py
experiments/runners/call_runner.py
experiments/runners/review_runner.py
experiments/runners/experiment_runner.py
```

第一轮只支持：

- 简单文档摘要任务
- 简单 JSON 抽取任务

这样就已经能跑完一条完整闭环。

------

## D. 最小评估与输出

必须先有：

```text
experiments/evaluators/task_success.py
experiments/evaluators/oracle_utility.py
experiments/analyzers/aggregate_results.py
```

第一轮先不急着画复杂图，只要能导出：

- 每任务成功/失败
- 选择了哪个服务
- utility 分是多少
- 最终 trust 怎么变

就够了。

# 五、落地顺序

这一部分最重要。
我建议你后面绝对不要并行乱写，而是严格按这个顺序推进。

------

## 第一步：跑通“一个模拟服务 + 一个任务 + 一次调用 + 一次评价”

目标：

- mock service 能被 AgentDNS 调用
- 响应头能拿到 `usage_id`
- 能提交评价
- trust 能更新

这一轮甚至不需要 discovery，手动指定服务都行。

这一步是实验基座最小闭环。

------

## 第二步：跑通“少量服务的 discovery + Top-K + 自动选择”

目标：

- 6 到 12 个服务预注册完成
- Agent 能走 `/client/discovery/search`
- 能按画像在 Top-K 里选服务
- 能跑 10 到 20 个任务实例

这一步会第一次把“服务选择”拉进来。

------

## 第三步：加入 oracle 日志与自动评估

目标：

- 每次调用都有隐藏 oracle 日志
- 能算：
  - 任务完成率
  - Oracle Hit
  - Oracle Utility

这一步后，你就不是单纯“系统跑通”，而是已经开始产出实验数据。

------

## 第四步：扩到两类任务、两类服务模板、多个 Agent 画像

目标：

- text_generation
- structured_extraction
- 至少 3 类 Agent 画像
- 30 到 50 个任务实例
- 开始有第一张可看的对比表

这一步非常重要，因为它会告诉你实验基座设计是否正确。

------

## 第五步：扩展到完整 8 大类服务与 54 个种子模板

目标：

- 192 个逻辑服务目录完成
- 8 大类 handler 完成
- 三应用域任务模板完成
- 24 个 Agent 全部可用

这一步后，主实验才能真正开始。

------

## 第六步：补 V2 与 V4

这是你当前后端必须补的小实现点。

也就是：

- V2：客观信号增强版
- V4：时间敏感 trust 版

这一步不需要大改系统架构，但它是主消融成立的必要前提。

------

## 第七步：正式跑四版本三阶段主实验

目标：

- 900 个任务实例 / 版本
- 三阶段
- 配对对比
- 自动汇总
- 自动出图表

这一步就是你论文主结果来源。

------

# 六、当前最小可行闭环到底长什么样

我建议你第一条真正落地的实验链，不要超过下面这个复杂度：

## 最小闭环场景

- 应用域：文档与知识处理
- 任务：单文档摘要
- 服务数量：6 个
- 服务类型：文本摘要 4 个 + JSON 抽取 2 个
- Agent 数量：2 个
  - 质量优先型
  - 成本优先型
- 消融版本：先只跑 V1 和 V3
- 任务实例数：10 到 20 个

## 这条最小闭环必须证明四件事

1. discovery 返回服务
2. Agent 能在 Top-K 里选
3. 调用后能自动生成 review 并更新 trust
4. 第二轮开始排序或选择出现变化

只要这一条跑通，后面整套实验基座基本就成立了。

------

# 七、哪些东西现在先不要做

为了避免你陷进去，我明确说几个现在不要先做的东西。

## 1. 不先做 192 个服务的全量命名与细节微调

先跑最小闭环，再扩。

## 2. 不先做所有 8 类 handler

先做 2 类，再扩到 4 类，再扩到 8 类。

## 3. 不先做演示 UI

当前第一优先级是实验，不是界面。

## 4. 不先做复杂图表系统

先保证 CSV 和基本统计表能出，再扩图。

## 5. 不先做个性化 trust

前面已经定了，不进入当前主线。

------

# 八、和当前后端联动时最需要注意的地方

这里我帮你提前标几个可能会踩坑的点。

## 1. `request_id / usage_id` 对齐

实验层 oracle 日志一定要能和：

- usage
- review
- task instance
  稳定关联起来。

## 2. stream / async 不要一开始就全上

第一轮闭环只做 sync。

## 3. 嵌套调用也不要一开始就上

第一轮先只做原子服务。

## 4. 评价生成规则必须和现在的 `reviews.py` 约束对齐

尤其是：

- `outcome` 不能和真实 final_state 冲突

## 5. 先不要碰生产性数据表结构

实验层优先用配置和日志解决，不轻易再改 ORM。
